# Intel Workbench Design

**Date:** 2026-03-09

## Goal

将当前项目监控页升级为一个面向个人技术提升的多页面工作台。核心定位不是通用聊天，也不是单页 dashboard，而是一个围绕项目变化、官方文档演进、技术分类聚合和项目知识问答的情报系统。

## Product Positioning

系统面向个人技术提升与项目跟踪，主价值是：

- 跟踪项目 release 变化
- 跟踪官方文档演进
- 将原始变化转成中文结构化结论
- 让用户可以基于这些积累进行问答、总结、比较和追问

因此整体定位是：

- 一部分是 `项目监控平台`
- 一部分是 `技术情报 / 变更分析工作台`
- AI 控制台定位为 `项目知识助手`

## Information Architecture

整体采用左侧固定导航 + 右侧主工作区。

一级页面分为：

### 1. 技术情报

首页，负责跨项目总览。

主要内容：

- 今日 / 本周高影响变化
- 热门技术域（网络、存储、调度、架构等）
- 最近新增固定结论
- 结构漂移或巡检告警

这是用户默认打开先看的页面。

### 2. 项目监控

按项目查看。

每个项目包含：

- 项目头部状态
- `ReleaseNote 区`
- `文档区`

其中：

- `ReleaseNote 区` 展示版本变化、兼容性风险、升级建议
- `文档区` 按技术分类板块聚合显示最新文档结论

如果项目没有文档源，则不展示 `文档区`。

### 3. AI 控制台

项目知识助手，不做代码执行，只做：

- 问答
- 搜索
- 总结
- 对比
- 追问

其回答来源于系统内部知识库，而不是纯裸模型聊天。

### 4. 配置中心

后台管理页，负责：

- 项目新增与编辑
- crawl profile 查看与调整
- 提示词配置
- 同步状态与 drift 告警

## Project Model

一个项目由两个主要输入组成：

- `GitHub URL`
- `官方文档 URL`

用户新增项目时只需要填写这两个地址和项目名。

系统自动生成：

- GitHub repo 标识
- Release 抓取器
- 文档发现结果
- crawl profile 草稿
- 默认分类规则

项目配置记录包括：

- `id`
- `name`
- `github_url`
- `docs_url`
- `repo`
- `enabled`
- `release_area_enabled`
- `docs_area_enabled`
- `sync_interval_minutes`
- `created_at`
- `updated_at`

## Release Knowledge Flow

Release 数据来源于 GitHub releases 与 changelog。

系统执行流程：

1. 抓取 release metadata
2. 必要时展开 changelog
3. 归一化为标准事件
4. 检查是否已分析过
5. 对新增或变化事件调用 LLM
6. 生成结构化中文结论
7. 存入知识层

ReleaseNote 区只展示最新三条，其他通过展开查看。

## Documentation Knowledge Flow

文档不能再只监控首页。

系统需要：

1. 从 `docs_url` 出发运行发现逻辑
2. 生成 crawl profile
3. 根据 profile 抓取章节页
4. 提取正文
5. 做技术域分类
6. 将相近页面聚合成分类结论
7. 对分类结论调用 LLM 生成中文说明

技术域包括：

- 网络
- 存储
- 调度
- 架构
- 安全
- 升级
- 运行时
- 可观测性
- 其他

目标不是逐页堆叠，而是：

- 一个技术域一组结论
- 每组默认展示最新三条
- 历史通过展开查看

## Agent-Assisted Discovery

文档站点结构不统一，因此项目新增时必须有 `发现 Agent` 参与。

发现 Agent 负责：

- 判断站点结构
- 找入口页、导航、版本入口、语言入口
- 判断允许抓取路径
- 识别不该抓的路径
- 生成 crawl profile 草稿

这份 crawl profile 将持久化，并允许在配置中心查看和修改。

## AI Console

AI 控制台不是通用聊天框，而是基于项目知识库的检索增强问答界面。

### Knowledge Layers

按优先级分三层：

1. 结构化结论层
   - release 中文结论
   - 文档分类结论
   - 建议动作
   - 技术标签
   - 固定结论

2. 文档片段层
   - 章节正文切片
   - 页面标题
   - URL
   - 技术分类

3. 原始来源层
   - release note 片段
   - changelog 原始片段
   - docs 原始页面片段

### QA Flow

1. 识别问题类型
2. 识别范围（全局 / 项目 / 分类 / 时间）
3. 先查结构化层
4. 不足时补文档片段层
5. 回答时附来源

### Output Structure

每次回答采用统一结构：

- 结论
- 关键依据
- 建议下一步
- 来源

## Frontend Principles

### 1. 首页不能是口号页

首页 Hero 必须专业、克制，不应该出现“版本变化直接讲人话”这类营销口号。

更合适的首页风格应是：

- 产品名
- 当前数据状态
- 当前范围（多少项目、多少 docs、多少结论）
- 明确入口（技术情报 / 项目监控 / AI 控制台）

### 2. 项目页强调层级

每个项目默认只突出最新三条。

- 其他条目折叠
- 单卡片默认紧凑态
- 详细说明通过卡片内展开

### 3. 多页面而不是单页堆叠

技术情报、项目监控、AI 控制台、配置中心应该分页面，不应该都塞在一个滚动页面里。

## Backend Requirements

必须满足：

- 新增项目时 docs 探测失败不能导致建项目失败
- 单条事件分析失败不能导致整轮同步失败
- 文档爬虫需要浅层优先
- 文档正文抽取优先 `<main>` / `<article>`
- 数据输出必须可追溯到原始来源

## Current Gaps to Address

当前系统已经具备基础骨架，但还存在这些未完成点：

- 仍然是单页工作区，未切成多页面导航
- AI 控制台尚未实现
- 文档分类质量还需进一步提升
- 配置中心只有项目新增的最小表单，还没有 profile 编辑和提示词页

## Recommendation

下一阶段实施顺序应为：

1. 切换为多页面工作台壳
2. 将项目监控页稳定为项目分区 + 最新三条 + 展开历史
3. 完成 AI 控制台的知识助手闭环
4. 完成配置中心的 crawl profile / prompt 编辑能力
