# Daily Project Intel Homepage Design

**Date:** 2026-03-10

## Goal

将当前 `技术情报` 首页从“按时间平铺的全局变化流”改造成“按项目汇总的每日情报首页”。首页的核心任务不是展示最近几条数据，而是每天为每个项目生成一张 `AI 今日摘要卡`，帮助用户快速判断最新动向和最值得关注的变化。

## User Intent

这个系统的主要价值是帮助用户掌握“最新动向”，而不是做一个静态报表页。因此首页必须优先回答三个问题：

- 每个项目今天最重要的变化是什么
- 这些变化为什么值得看
- 如果只看 1 到 3 个依据，应该先看哪几个

## Current Problems

当前首页存在几个明显问题：

- `高影响变化` 实际上只是把所有项目条目摊平后按时间排序，再取最近三条
- 首页没有项目级聚合层，导致用户看到的是零散结论，不是可消费的项目摘要
- 文档分类和来源覆盖更多是统计视图，对“今天该看什么”帮助有限
- LLM 只参与单条事件分析，没有参与首页排序和项目级汇总
- 同步是按小时轮询，首页没有“按天生成日报”的概念

## Chosen Approach

采用“每日项目摘要层”方案。

系统保留现有 `events + analyses` 事件知识层，但在其上新增一层 `daily project summary`：

1. 先抓取和分析单条 release / docs 事件
2. 再按项目聚合最近一天或最近一个观察窗口内的变化
3. 让 LLM 为每个项目生成一张首页摘要卡
4. 首页只消费项目摘要卡，而不是直接消费事件流

这个方案比“重新调排序规则”更适合首页，因为它让首页的数据形态直接贴合用户目标。

## Data Architecture

首页的目标数据链路定义为：

`projects` -> `events` -> `analyses` -> `daily_project_summaries` -> `dashboard.homepage_projects`

各层职责如下：

- `projects`
  - 项目注册表，定义项目边界
- `events`
  - 原始 release / docs 标准化事件
- `analyses`
  - 单条事件级中文结构化结论
- `daily_project_summaries`
  - 新增的项目日报层，按“项目 + 日期”聚合
- `dashboard.homepage_projects`
  - 面向首页的视图层结果，按重要度排序

## Daily Summary Model

每个项目的日报记录建议包含：

- `project_id`
- `date`
- `summary_zh`
- `headline`
- `importance`
- `reason`
- `evidence_ids`
- `evidence_items`
- `updated_at`

其中：

- `summary_zh` 是首页主摘要
- `headline` 是卡片标题
- `importance` 用于排序
- `reason` 解释为什么今天值得看
- `evidence_items` 只保留 1 到 3 条依据，供首页直接展示

## Ranking Rules

首页不再按纯时间排序，而是按项目摘要卡排序。

排序优先级：

1. `importance`
2. 是否存在 `action_items`
3. 证据中最高 `urgency`
4. `updated_at`
5. 项目名

如果某个项目当天没有新增高价值变化：

- 仍保留项目卡
- 摘要可退化为“今日无显著变化，建议关注最近固定结论”

这样首页不会因为当天抓不到新数据而出现项目缺位。

## Homepage Layout

首页改成单列项目卡流，避免继续做全局混排。

每张项目卡包含：

- 项目名
- 今日标题
- 一段 AI 摘要
- 值得看的原因
- 1 到 3 条依据
- 最近更新时间
- 跳转到项目页 / 来源页

页面顶部保留轻量总览，但弱化“统计看板”角色，强调“今日情报首页”定位。

## Update Strategy

系统保留现有同步机制，但新增首页摘要生成策略：

- 原始事件抓取和单条分析：继续由同步任务驱动
- 项目日报生成：每天最多刷新一次
- 用户手动点击 `立即同步` 时，如果当天摘要未生成或证据已变化，则立即重算

默认更新节奏可先设置为每天一次，后续再暴露为配置项。

## Desktop Start/Stop Flow

用户不需要常驻后台服务，只需要桌面一键开关。

采用两个 macOS `.command` 文件：

- `启动 Intel Workbench.command`
- `停止 Intel Workbench.command`

启动脚本负责：

- 启动后端服务
- 启动前端服务
- 写入 PID 文件
- 打开浏览器到本地地址

停止脚本负责：

- 读取 PID 文件
- 停止前后端进程
- 清理运行状态文件

## Error Handling

- 项目日报生成失败时，不影响原始同步完成
- 首页如果没有日报数据，应回退到明确空态，而不是继续使用旧的“全局时间流”伪装成摘要
- 某个项目摘要失败时，只影响该项目卡，不阻断其他项目

## Testing Strategy

后端需要覆盖：

- 日报生成逻辑
- 首页 API 返回项目摘要卡
- 当天无新增变化时的回退摘要

前端需要覆盖：

- 首页默认渲染项目摘要卡
- 每张卡展示摘要和 1 到 3 条依据
- 不再出现旧的“最近三条时间流高影响变化”作为首页主结构

桌面脚本需要验证：

- 启动后能访问前端页面
- 停止后相关进程退出
- 重复执行不会产生脏状态
