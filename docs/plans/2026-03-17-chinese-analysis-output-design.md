# 分析与日报中文输出设计

## 目标

把面向用户的分析内容统一收口为中文输出。

本次约束明确如下：
- 文档原文、页面标题、diff 原文可以保留英文
- 系统生成的分析结果必须是中文
- 日报卡片里的标题、摘要、依据文案必须是中文
- 当历史数据或模型返回结果不满足要求时，系统要自动回退到简短可读的中文，而不是直接把英文或占位文案暴露给用户

## 范围

本次只处理两条用户明确要求的内容链路：
- 文档台中的分析结果
- 首页/日报中的摘要内容

暂不处理：
- 页面原文内容
- diff 原文块
- 站内所有英文 UI 标签以外的原始上游字段

## 问题

当前存在两类问题：

1. 文档分析会漏出英文
- `summary_zh`
- `doc_summary`
- `reading_guide`
- `diff_highlights`
- `details_zh`

这些字段理论上应该是中文，但历史数据、占位回退或页面摘要回退会把英文内容带到前端。

2. 日报内容会漏出英文
- `headline`
- `summary_zh`
- `reason`
- `evidence_items[].title_zh`
- `evidence_items[].summary_zh`

这些字段来自历史分析或默认拼接逻辑，当前在中文字段缺失时可能直接吃到英文标题和英文摘要。

## 方案

采用“后端统一中文兜底，前端停止拿原文充分析”的方案。

### 1. 后端统一判断“是否为可接受中文分析”

增加一个轻量判断函数，用于识别：
- 空字符串
- 已知占位文案
- 明显不满足中文输出要求的英文文本

判断原则：
- 不是做语言学检测，只做产品兜底
- 只要一个字段几乎全是 ASCII/英文词，就视为不合格
- 混合文本里如果有足够中文，可以保留

### 2. 文档分析 API 统一输出中文

在 `backend/app.py` 的文档事件聚合链路中，对下面字段统一做中文清洗：
- `title_zh`
- `summary_zh`
- `details_zh`
- `doc_summary`
- `diff_highlights`
- `reading_guide`
- `impact_points`
- `action_items`

回退策略：
- 优先保留已有中文分析
- 若字段不合格，则生成简短中文模板，例如：
  - `该页面暂无可用中文解读，请先看页面摘要与 diff。`
  - `建议先看 <页面名> 的页面变化。`
  - `当前证据不足，建议结合页面摘要继续判断。`
- 不再把 `page.summary` 的英文内容直接塞进这些分析字段

### 3. 日报构建强制中文

在 `backend/daily_summary.py` 中：
- 证据项的 `title_zh`、`summary_zh` 只允许来自中文分析字段或中文模板
- `headline`、`summary_zh`、`reason` 不再直接吃英文标题
- 当证据只有英文标题时，改成中文模板，例如：
  - `<项目名> 今日重点`
  - `今天检测到新的项目变化，建议查看最新中文解读。`

### 4. 前端停止拿英文原文冒充分析

在 `src/components/DocsWorkbenchPage.jsx` 中：
- 页面列表 teaser 不再回退到英文 `page.summary`
- 右侧 `变化 / 影响 / 建议` 不再把英文页面摘要当成分析结果
- 当后端给不出中文分析时，前端仅显示中文兜底文案

首页日报卡也只显示中文分析字段，不额外拼英文兜底。

## 数据兼容

新规则需要兼容已有历史数据。

做法：
- API 层先做运行时清洗，保证页面立刻变中文
- 如有必要，再对历史 `analyses.json` 和日报缓存做一次本地修复
- 不要求立即全量重跑同步

## 测试

需要覆盖三类行为：

1. 文档分析字段是英文时
- API 应返回中文兜底
- 前端不显示英文分析文案

2. 日报证据项是英文时
- 构建结果应改为中文标题和中文摘要
- 首页卡片不展示英文分析内容

3. 原文仍可保留英文
- 页面标题、原始 diff、新增删除块可继续展示英文
- 只验证系统生成字段被中文化

## 结果预期

完成后，用户在产品里看到的：
- 文档分析结果是中文
- 日报摘要是中文
- 原文仍然可以是英文
- 即使历史数据脏，也不会再把英文分析直接暴露出来
